# MCP 스펙 2026-07-28 RC — stateless + first-class extensions

## TL;DR
[원본 §Introduction →](2026-05-27_mcp_spec_2026_07_28_rc_source.md#introduction)

- [[MCP|Model Context Protocol]] 스펙 **2026-07-28 릴리스 후보** — 메인테이너(David Soria Parra·Den Delimarsky)가 "**출시 이후 최대 개정**"으로 규정. 블로그 2026-05-21.
- 4대 축: **① stateless 프로토콜** ② **first-class extensions** ③ **OAuth/OIDC 인증 강화** ④ **feature lifecycle 정책**(12개월 deprecation 보장).
- 가장 큰 변화 = **세션 제거**. `initialize`/`initialized` 핸드셰이크 + `Mcp-Session-Id` 헤더 삭제 → 요청이 self-contained(`_meta`에 버전·클라이언트 정보) → **sticky session 불필요, round-robin LB로 충분**.
- breaking 변화 다수: Roots/Sampling/Logging **deprecated**, Tasks가 코어→**extension 분리**(마이그레이션 필요), error code `-32002`→`-32602`.
- 일정: **RC lock 5/21**, **final 7/28**, SDK 검증 윈도우 **10주**, Tier 1 SDK는 윈도우 내 지원.

## ① Stateless 프로토콜 — 세션이 사라졌다
[원본 §A Stateless Protocol →](2026-05-27_mcp_spec_2026_07_28_rc_source.md#a-stateless-protocol)

- **Before (2025-11-25)**: `initialize` 핸드셰이크가 session ID 반환 → 이후 모든 요청이 특정 서버 인스턴스에 핀.
- **After (2026-07-28)**: 요청마다 `_meta`에 클라이언트 정보 → **아무 인스턴스나 처리 가능**.
  ```
  POST /mcp HTTP/1.1
  MCP-Protocol-Version: 2026-07-28
  Mcp-Method: tools/call
  Mcp-Name: search
  ```
- **인프라 임팩트**: "sticky session + 공유 session store + 게이트웨이 deep packet inspection이 필요했던 원격 MCP 서버가 이제 평범한 round-robin LB 뒤에서 동작".
- **stateless 프로토콜 ≠ stateless 앱**: 프로토콜이 세션 상태를 관리 안 할 뿐, 앱은 `basket_id` 같은 **명시적 handle**을 tool call 간 전달 → 상태가 모델에게 **숨겨지지 않고 보이게** 됨.
- **server→client 요청 재구성**: SEP-2260(서버 개시 요청은 active client 요청 처리 중에만 허용), **SEP-2322**(Multi Round-Trip — SSE 스트림 대신 `InputRequiredResult`로 `inputRequests`/`requestState` 반환, 클라이언트가 `inputResponses`로 재시도).
- **routable/cacheable/traceable**: SEP-2243(`Mcp-Method`/`Mcp-Name` 헤더로 body 검사 없이 라우팅), SEP-2549(결과에 `ttlMs`/`cacheScope` — HTTP `Cache-Control` 모델), SEP-414(W3C Trace Context 분산 추적).

## ② Extensions가 first-class로
[원본 §Extensions Become First-Class →](2026-05-27_mcp_spec_2026_07_28_rc_source.md#extensions-become-first-class)

- **SEP-2133**: extensions 프레임워크 공식화 — reverse-DNS 식별자, capability map 협상, 별도 `ext-*` 저장소 + 위임 메인테이너, **독립 버전 관리**.
- **MCP Apps (SEP-1865)**: 서버가 **sandboxed iframe**에 렌더되는 인터랙티브 HTML UI 제공. tool이 UI 템플릿을 사전 선언(prefetch + 보안 검토). UI 액션도 직접 tool call과 **동일한 JSON-RPC** → audit trail 유지.
- **Tasks → extension 강등**: 2025-11-25 실험적 코어 기능이던 Tasks가 프로덕션 재설계 필요로 **공식 extension으로 분리**. 새 lifecycle — 서버가 `tools/call`에 task handle 응답, 클라이언트가 `tasks/get`·`tasks/update`·`tasks/cancel`로 구동. **task 생성은 서버 주도**, `tasks/list`는 세션 없이 안전치 않아 **제거**. ⚠ 기존 실험 API 코드는 **마이그레이션 필수**.

## ③ 인증 강화 — OAuth 2.0 / OIDC 정렬
[원본 §Authorization Hardening →](2026-05-27_mcp_spec_2026_07_28_rc_source.md#authorization-hardening)

6개 SEP로 OAuth 2.0·OpenID Connect 관행에 정렬:

| SEP | 내용 |
|---|---|
| SEP-2468 | 클라이언트가 `iss` 파라미터 검증 (RFC 9207) |
| SEP-837 | DCR 시 OIDC `application_type` 선언 |
| SEP-2352 | 자격증명을 발급 authz 서버에 바인딩 (서버 이전 시 재등록) |
| SEP-2207 | OIDC 서버 refresh token 요청 문서화 |
| SEP-2350 | step-up 시 scope 누적 명확화 |
| SEP-2351 | `.well-known` discovery suffix 명확화 |

## ④ Roots / Sampling / Logging deprecated
[원본 §Roots, Sampling, and Logging Are Deprecated →](2026-05-27_mcp_spec_2026_07_28_rc_source.md#roots-sampling-and-logging-are-deprecated)

새 lifecycle 정책(SEP-2577)으로 3기능 deprecated:

| 기능 | 대체 |
|---|---|
| Roots | tool 파라미터 / resource URI / 서버 설정 |
| Sampling | LLM 프로바이더 API 직접 통합 |
| Logging | stdio는 stderr / 구조적 관측은 OpenTelemetry |

- annotation-only — 메서드/타입은 **제거 전 최소 12개월** 동안 계속 동작.
- **Sampling deprecation 주목**: 서버가 클라이언트 LLM을 빌려 쓰던 패턴 폐기 → 프로바이더 API 직접 통합 권장. 에이전트 설계 영향.

## ⑤ tool에 JSON Schema 2020-12 전면 지원
[원본 §Full JSON Schema 2020-12 for Tools →](2026-05-27_mcp_spec_2026_07_28_rc_source.md#full-json-schema-2020-12-for-tools)

- **SEP-2106**: tool `inputSchema`/`outputSchema`를 JSON Schema 2020-12 전면 지원으로 격상. input은 root `type: "object"` 유지하되 **composition(`oneOf`/`anyOf`/`allOf`)·conditional·reference** 허용. output은 제약 없음.
- **error code 변경 (SEP-2164)**: 리소스 없음 에러가 커스텀 `-32002` → JSON-RPC 표준 **`-32602` Invalid Params**. ⚠ 리터럴 `-32002` 매칭하는 클라이언트는 업데이트 필요.

## ⑥ 진화 프레임워크 — breaking-change 없는 발전
[원본 §How the Protocol Evolves From Here →](2026-05-27_mcp_spec_2026_07_28_rc_source.md#how-the-protocol-evolves-from-here)

3개 거버넌스 SEP:
1. **Feature Lifecycle Policy**: Active→Deprecated→Removed, **최소 12개월** deprecation 윈도우.
2. **Extensions Framework**: 신기능은 opt-in extension으로 출시 → 안정화 후 (그럴 경우에만) 스펙 편입.
3. **Conformance Requirement (SEP-2484)**: Standards Track SEP는 conformance suite 시나리오 매칭 없이 Final 불가. **SDK tier 시스템**이 공식 SDK를 suite 대비 점수화.

## 실무 영향 (PAB 관점)

- **서버 운영 단순화**: stateless 덕에 MCP 서버를 stateless 컨테이너로 수평 확장 + 평범한 LB. 세션 affinity 인프라 제거.
- **마이그레이션 부채**: ① Tasks 실험 API → 새 lifecycle ② `-32002` 매칭 코드 → `-32602` ③ Roots/Sampling/Logging 사용처 → 대체 경로 (12개월 유예).
- **에이전트 패턴**: Sampling deprecated → 서버가 클라이언트 LLM 빌리는 설계 지양, 프로바이더 직접 통합. MCP Apps로 서버 제공 UI 가능(sandboxed iframe).
