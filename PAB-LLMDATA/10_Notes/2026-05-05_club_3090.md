# club-3090 — RTX 3090에서 LLM 서빙하는 두 가지 경로

## TL;DR
[원본 §TL;DR — what this is →](2026-05-05_club_3090_source.md#tldr--what-this-is)

- **두 경로, 워크로드 깨지는 지점에 따라 선택**:
  - 🏎 **[[vLLM]] dual** — 최대 throughput. 최대 **127 TPS**(코드, [[DFlash]]) 또는 **262K에서 4 동시 스트림**(turbo). 비전·툴·[[MTP]]·스트리밍 풀 스택.
  - 🛡 **[[llama.cpp]] single** — 최대 robustness. 한 장의 3090에서 **262K 컨텍스트** 풀. prefill cliff 없음, 25K 토큰 툴 응답·90K needle 통과. 느림(~21 TPS)이지만 실제 툴 사용 에이전트에서 안 죽음.
- **검증된 docker-compose 설정**으로 양쪽 경로 모두 `localhost:8020`에서 OpenAI 호환 API 즉시 가동.
- 멀티 엔진 지원: [[vLLM]](풀 기능), [[llama.cpp]](최대 ctx + 견고함), [[SGLang]](현재 blocked, watch list).
- **모델 무관**(model-agnostic) 설계. 현재 [[Qwen3.6-27B]] 우선 — 구조가 확장 가능해 신모델 추가 시 `models/<name>/` 하위에 동일 패턴으로 들어옴.

## 두 엔진 경로의 의미
[원본 §TL;DR — what this is →](2026-05-05_club_3090_source.md#tldr--what-this-is)

3090 24GB 한 장 또는 두 장 환경에서 27B급 LLM을 서빙할 때, **워크로드가 어디서 깨지는가**가 엔진 선택의 기준이다.

| 측면 | vLLM dual | llama.cpp single |
|---|---|---|
| 우선순위 | **throughput** (TPS) | **robustness** (안정성) |
| 카드 수 | 2× 3090 | 1× 3090 |
| 최대 ctx | 262K + 4 동시 스트림 (turbo) | 262K (단일 스트림) |
| 측정 TPS | 89~127 (코드, DFlash) | ~21 |
| 비전·툴 | ✅ 풀 스택 | ⚠️ 제한적 |
| prefill cliff | (모델별 패치 필요) | 없음 (검증됨) |
| 운영 형태 | docker compose | 직접 실행 (Docker 불필요) |

vLLM은 throughput·기능 풀 스택이지만 모델별 [[Genesis patches]]·DeltaNet cliff 같은 패치를 따라가야 한다. llama.cpp는 느리지만 prefill cliff·25K 툴 응답·90K needle 같은 실전 시나리오에서 죽지 않는다. 둘 중 하나를 고르는 것이 아니라 **워크로드별로 둘 다 가지고 있는 것**이 이 저장소의 권장 구도.

## 지원 모델 — 현재 Qwen3.6-27B만
[원본 §Supported models →](2026-05-05_club_3090_source.md#supported-models)

현재 production-ready 모델은 [[Qwen3.6-27B]] 단 1종. 1×/2× 3090에서 vLLM·llama.cpp 모두 동작 (SGLang은 blocked). 비전·툴·MTP n=3·262K ctx 지원. vLLM dual에서 89/127 TPS, llama.cpp single에서 prefill cliff 없는 풀 262K.

> 신모델(예: GLM-4.6)은 같은 `models/<name>/` 패턴으로 추가될 예정.

## Quick Start (Qwen3.6-27B + vLLM 기준)
[원본 §Quick start →](2026-05-05_club_3090_source.md#quick-start-for-the-current-model--qwen36-27b-on-vllm)

7단계 워크플로우:

```bash
# 1. clone
git clone https://github.com/noonghunna/club-3090.git && cd club-3090

# 2. 모델 다운 + SHA 검증 (~20GB, Genesis 패치도 함께)
bash scripts/setup.sh qwen3.6-27b

# 3. 엔진/카드/워크로드 선택 wizard 또는 --variant 직접 지정
bash scripts/launch.sh
#   bash scripts/launch.sh --variant vllm/default       # 단일 카드 채팅
#   bash scripts/launch.sh --variant vllm/dual          # 듀얼 262K + 비전
#   bash scripts/launch.sh --variant llamacpp/default   # 단일 카드 262K, no cliffs

# 4. sanity test
curl -sf http://localhost:8020/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"qwen3.6-27b-autoround","messages":[{"role":"user","content":"Capital of France?"}],"max_tokens":200}'

# 5. canonical bench
bash scripts/bench.sh

# 6. 변형 전환 (wizard 재진입 없이)
bash scripts/switch.sh vllm/long-vision

# 7. 스택 따라 업데이트 (Genesis pin bump, 새 compose, vendored patch 갱신)
bash scripts/update.sh
```

`launch.sh`는 `switch.sh`(이전 컨테이너 down → 새 컨테이너 up) + `verify-full.sh`(헬스 체크)를 자동 호출. 클라이언트 통합(Open WebUI·Cline·Cursor·Python `openai` SDK 등)은 `docs/EXAMPLES.md` 참조.

## 저장소 구조 철학 — 엔진·모델·스크립트 분리
[원본 §How this is structured →](2026-05-05_club_3090_source.md#how-this-is-structured)

확장 가능한 구조를 위해 3축으로 명확히 분리:

1. **엔진과 하드웨어는 일반(general)** — `docs/`의 문서는 모델 무관 적용. vLLM은 Qwen이든 GLM이든 Llama든 동일하게 동작 → 엔진 문서는 한 번만 작성.
2. **모델은 특수(specific)** — `models/<name>/` 하위에 quant·quirk·권장 config·엔진별 레시피 위치. 신모델 추가 = 같은 내부 패턴의 새 서브디렉토리 1개.
3. **스크립트는 공유하되 모델 인지(shared but model-aware)** — `bash scripts/setup.sh qwen3.6-27b`가 올바른 모델·패치를 가져옴. 신모델 추가 시 `bash scripts/setup.sh glm-4.6`로 동일 스크립트가 처리.

> 최상단 README는 "stack docs"만, 모델 디테일은 전용 서브디렉토리에. 이 분리가 스택이 커져도 유지 가능하게 만든다.

## 요구사항
[원본 §What you'll need →](2026-05-05_club_3090_source.md#what-youll-need)

| 항목 | 비고 |
|---|---|
| 1× 또는 2× RTX 3090 (24GB) | 4090·A6000 등 더 큰 Ampere/Ada도 OK. 12GB 카드는 27B급 못 올림. |
| Linux (Ubuntu 22.04+ 검증) | vLLM은 Linux + CUDA only. llama.cpp는 macOS/Windows OK이지만 레시피는 Linux 경로 가정. |
| Docker + NVIDIA Container Toolkit | vLLM 용. llama.cpp는 Docker 없이 동작. |
| NVIDIA driver 580.x+ | vLLM nightly의 CUDA 13 런타임용. |
| ~30 GB 디스크 (모델당) | 다중 모델 사용 시 더 필요. |
