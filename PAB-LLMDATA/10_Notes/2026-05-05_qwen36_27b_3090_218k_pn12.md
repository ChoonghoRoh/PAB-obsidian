# Qwen3.6-27B on 1× RTX 3090 — 218K context + PN12 fix (Reddit follow-up)

## TL;DR
[원본 §Selftext — Current results →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#selftext--current-results)

- [[club-3090]] 운영자 [[AmazingDrivers4u]]가 r/LocalLLaMA에 올린 **follow-up 게시글** — 이전 125K @ 85 TPS 빌드를 **218K @ 50–66 TPS**로 확장. throughput을 일부 양보하고 컨텍스트·안정성을 가져온 절충.
- 단일 [[RTX 3090]] 24GB에서 **~218K context @ 50/66 TPS** (text, narr/code), **~198K + vision @ 51/68 TPS**.
- ~25K-token 툴 응답이 일관되게 OOM으로 죽던 문제가 **[[PN12]] anchor drift fix** 이후 해소.
- 댓글에서 발굴된 핵심 통찰: PN12 패치가 [[vLLM]] `torch.compile` inductor 경로에서 **여전히 우회**된다 — 합성 25K 테스트는 통과하지만 OpenCode 같은 실제 워크로드는 깨짐. 워크어라운드 3종 도출.

## 무엇이 바뀌었나 — PN12 anchor drift 수정
[원본 §What changed →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#what-changed)

- [[Genesis patches]]의 [[PN12]] 패치가 **vLLM dev205+에서 적용되지 않고 있었음**:
  - `apply_all`은 성공으로 보고
  - 그러나 실제 코드 경로는 변경되지 않음 → **anchor drift** (패치 anchor가 vLLM 신버전에서 이동)
- anchor drift 수정 후 → tool-prefill OOM 사라지고 더 높은 컨텍스트 설정 사용 가능.
- Fix: `Sandermage/genesis-vllm-patches` PR #13.
- **교훈**: "패치 적용됨" 보고와 "패치 동작" 검증은 별개. 후자는 동작 차이로만 확인 가능.

## 두 차원 동시 최적화 — context × throughput × 툴 안정성
[원본 §What were optimizing for →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#what-were-optimizing-for)

목표는 단일 [[RTX 3090]]에서 다음 셋을 *동시에* 끌어올리기:

- **고 컨텍스트** (200K+)
- **사용 가능한 throughput** (TPS 절대값보다 "써먹을 수 있느냐")
- **안정적 [[tool-agent]] 워크로드** (25K+ 토큰 응답에서도 안 죽음)

→ 한 차원만 보면 더 높은 수치도 가능하지만, 실전 에이전트는 셋 다 무너지면 가치 없음.

## 메모리 cliff — 50–60K 단일 GPU 잔존
[원본 §Notes / limitations →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#notes--limitations)

- 단일 GPU 단일 프롬프트 워크로드에서 **~50–60K 부근에 두 번째 메모리 cliff** 잔존.
- [[tensor parallelism]] (예: 2× 3090) 사용 시 해당 cliff는 **나타나지 않음**.
- 결과는 [[quantization]] + 설정에 강하게 의존.
- 자세한 분석: `noonghunna/club-3090/docs/CLIFFS.md`.

## 이전 빌드 검증 — G2 vLLM Stack benchmark
[원본 §G2 vLLM Stack benchmark by ImportantQuote1180 →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#g2-vllm-stack-benchmark-by-importantquote1180)

댓글러 `Important_Quote_1180`가 이전 빌드(125K @ 82 TPS)의 실제 운영 수치 보고:

| 항목 | 값 |
|---|---|
| 모델 | qwen3.6-27b-autoround-int4 ([[AutoRound INT4]]) |
| vLLM | nightly dev21, port 8020 |
| 컨텍스트 | 125K |
| KV cache | [[TurboQuant 3-bit NC]] |
| Speculative decoding | [[MTP]] (3 draft tokens) |
| Cudagraph mode | **PIECEWISE** (FULL 모드는 MTP와 함께 출력 garble) |
| GPU 사용률 | 23.1/24.5 GB (97%), 348W/350W, 66°C, 98% 활용 |

**Sustained TPS**: 67–89 (워크로드 형태 의존).
실측 (2026-04-26): 100토큰 82.4 tok/s, 400토큰 82.1 tok/s, 800토큰 71.3 tok/s, TTFT 0.3–0.6s.

**핵심 트레이드오프**: PIECEWISE는 이론 FULL 대비 **15–20% 처리량 손실**, 그러나 FULL은 [[MTP]]와 결합 시 garbled 반복 출력 → "82 tok/s clean output > 108 tok/s garbled output"이라는 명시적 선택.

## 5090 / 듀얼 3090 / WSL2 비교
[원본 §5090 results by OptimalBass5246 and Dany0 →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#5090-results-by-optimalbass5246-and-dany0)

- **RTX 5090** (32GB): 160+ TPS, 풀 256K context. WSL에서도 150+ TPS, Ubuntu 네이티브에서 160+. 비전도 동작 ([[CobraPhil/qwen36-27b-single-5090]] 레시피 검증).
- **혼합 워크로드 실측**: 단순 코딩 prompt 디코드 only 160 TPS, 45–200K 실제 코딩+제너럴리스트 워크로드는 ~90 TPS. quantization 품질은 ud Q4_k_xl 정도 체감.
- **반복 루프 회피**: repetition penalty 1.04 + presence penalty 0.1 → 루프 감소(플라시보 가능성 단서 함께 명시).
- **80–120K 이후 정지/poison token** 빈발 — [[fp8]] 버전이 이 영역에서 더 안정. 회피책: 대화 이력을 "in-distribution"으로 깨끗하게 유지.
- ⚠ **[[WSL2]] + vLLM 한계 사례**: 단일 4090 사용자가 윈도우 매니저와 함께는 어떤 설정으로도 128K 못 도달, 최선 48K. 다른 댓글러도 WSL2에서 OOM 일관 보고.

> WSL에서 시도하는 경우 **Ubuntu 네이티브 대비 성능 손실**을 미리 감안. 클럽 3090 스크립트 자체가 WSL 검증 환경이 아님.

## OpenCode prefill OOM — torch.compile inductor 우회 이슈
[원본 §OpenCode prefill OOM diagnosis by VolandBerlioz →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#opencode-prefill-oom-diagnosis-by-volandberlioz)

`VolandBerlioz`가 PN12 fix 적용 *이후*에도 [[OpenCode]] 워크로드에서 prefill OOM을 재현하고 Codex로 분석한 진단:

**증상**: 모델 부팅·요청 수락 정상 → 실제 OpenCode 긴 컨텍스트 요청에서 `chunked prefill` 중 CUDA OOM → HTTP 500 + vLLM 프로세스 종료.

**OOM 발생 지점**:
```
inductor_cache/...py:1208
buf9 = empty_strided_cuda((s18, 17408), (17408, 1), torch.float16)
torch.OutOfMemoryError: Tried to allocate 138.00 MiB
```

**메모리 상태**: GPU 23.56 GiB 중 free 131.75 MiB, PyTorch allocated 22.59 GiB, 요청 138 MiB → **수 MiB 차이로 실패**.

**할당량 추정**:
```
max_num_batched_tokens × intermediate_size × fp16
≈ 4128 × 17408 × 2 bytes ≈ 138–144 MiB
```

**중요한 단서**: `pn12_ffn_pool_anchor_fix`도, `fa_max_seqlen_clamp`도 모두 "applied" 로그 출력. 그런데도 OOM. 즉 **패치는 붙었는데 효과가 안 나는 상태**.

→ [[Cliff 1]] FA2 softmax_lse 경로 아닌 **FFN intermediate activation 할당 단계** 문제.

## 워크어라운드 3종 — 운영자가 댓글로 정리
[원본 §PN12 inductor path analysis by AmazingDrivers4u →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#pn12-inductor-path-analysis-by-amazingdrivers4u)

운영자([[AmazingDrivers4u]])가 위 진단을 받고 정확한 원인을 댓글로 공개 + 워크어라운드 3종 (`club-3090/issues/16`):

**근본 원인**: PN12은 eager 모드 `SiluAndMul.forward_cuda`를 패치 — 그런데 vLLM의 `torch.compile` inductor가 SiluAndMul op를 **인라인**해서 패치된 메서드를 *호출하지 않음*. 즉 compile path에서는 PN12 풀이 우회됨. 합성 25K 테스트는 우연히 eager 경로를 타는 shape이라 통과, OpenCode 29K 시스템+툴 혼합 prefill은 inductor 경로 shape이라 OOM.

| # | 워크어라운드 | 비용 | 효과 |
|---|---|---|---|
| 1 | **`tools-text.yml` 그대로 사용** | 없음 (이미 동작) | fp8 KV는 [[PN8]] 사용 → compile path도 도달. 75K context로 30K OpenCode prefill 충분 |
| 2 | `long-text.yml`/`long-vision.yml`에 **`--enforce-eager`** 추가 | TPS **20–30% 감소** | 모든 forward를 eager Python으로 강제 → PN12 풀 안정 적용. 218K/198K context 보존 |
| 3 | `--gpu-memory-utilization`을 **0.94로 하향** | KV pool 축소 → 실효 max_model_len ~150K | activation headroom ~250 MiB 확보 |

**진짜 fix**는 inductor-pass 레벨 개입 또는 torch.compile-aware sidecar — 큰 작업이라 issue #16에서 별도 트래킹.

> 교훈: **"패치 applied" 로그가 곧 효과 보장이 아니다.** vLLM의 torch.compile inductor 경로는 eager 메서드 패치를 인라인으로 우회한다. compile path 진입을 확인할 검증이 별도로 필요.

## 한계 / 미해결
[원본 §Notes / limitations →](2026-05-05_qwen36_27b_3090_218k_pn12_source.md#notes--limitations)

- 단일 GPU 50–60K **두 번째 cliff** 잔존 (TP 시 해소).
- PN12 inductor 우회 — 워크어라운드만 존재, 진짜 패치는 `club-3090/issues/16`에서 작업 중.
- 80–120K 이후 [[poison token]] / 정지 / 루프 가능성 — quantization 한계, 회피는 대화 이력 정돈.
- [[WSL2]] 환경은 동등 성능 미보장 — 호스트 OS의 메모리·드라이버 영향이 크다.
- 35B A3B + LoRA + RAM offload 같은 다른 형태는 아직 별도 config 없음.
