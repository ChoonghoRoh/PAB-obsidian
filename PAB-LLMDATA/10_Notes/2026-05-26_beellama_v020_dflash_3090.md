# BeeLlama.cpp v0.2.0 — DFlash로 단일 RTX 3090 생성 4.4~4.93x

## TL;DR
[원본 §Resources →](2026-05-26_beellama_v020_dflash_3090_source.md#resources)

- [[BeeLlama.cpp]] = Anbeeld의 **[[llama.cpp]] 포크**. 단일 [[RTX 3090]](24GB)에서 GGUF 추론 속도를 짜내는 데 집중. v0.2.0 (2026-05-22), r/LocalLLaMA 게시.
- 핵심 = **DFlash [[speculative decoding]]** — draft 모델이 target의 hidden state에 cross-attend해 토큰 제안, target이 검증. (메커니즘 상세는 [GitHub README](https://github.com/Anbeeld/beellama.cpp))
- 벤치(RTX 3090, baseline llama.cpp b9275 CUDA 13.1): **Qwen3.6-27B 163.9 tps median (4.40x)**, **Gemma4-31B 177.8 tps (4.93x)**. 거의 5배.
- **핵심 비교 — 같은 하드웨어 [[MTP]] spec-decode는 1.7~1.9x뿐**. DFlash가 MTP를 2배 이상 앞섬(아래 §DFlash vs MTP). [[2026-05-07_qwen36_dual_3090_25_to_283_tps|Ollman 듀얼 3090]]이 의존한 게 바로 MTP n=5.
- ⚠️ **가속은 token generation 전용**. prompt processing/prefill은 baseline **0.99x** (Qwen 1229.5→1214.4 tok/s) — 단, mainline MTP(0.95x)보단 빠름. 생성 토큰 많은 워크로드(코드/agentic)에서만 큰 이득.
- 위치: **실험 도구** — 멀티 GPU 미지원, long-context에서 acceptance 급락. 단 출력 정확도 손실 없음(target이 전 draft 검증).

## DFlash vs MTP — acceptance "2개 숫자"의 함정
[원본 §Comments →](2026-05-26_beellama_v020_dflash_3090_source.md#comments)

댓글에서 가장 중요한 기술 토론. acceptance 컬럼은 **두 숫자** = `제안 draft 중 채택 비율 / 채택 draft가 최종 생성에 기여한 비율`.

- **MTP의 acceptance가 더 높아 보이는 착시**: MTP는 draft-n-max **3 고정**(보수적 draft) → 첫 숫자(채택률) 높음(92%). DFlash는 draft-n-max **16 + profit 기반 adaptive controller** → 많이 던지고 일부 실패 → 첫 숫자 낮음(67%).
- **Anbeeld 핵심 반박**: 의미 있는 건 **두 번째 숫자**(채택 draft→최종 생성 기여). **모든 벤치에서 DFlash가 MTP보다 높음**. draft가 v0.1.2부터 싸졌기에 "많이 던지는" 게 이득.
- **반대 관점(laul_pogan)**: long context에선 DFlash acceptance 급락 — 28K multi-turn에서 **24.9%** vs MTP **71.9%**. DFlash는 short-burst(1K, 4x+)에서 압승하지만 **200k 롤링 세션은 MTP가 우세**할 수 있음. 실제 사용 context 깊이에서 acceptance 측정 후 선택 권장.
- **VRAM**: DFlash 오버헤드 27B-31B dense에서 **1-2GB뿐** (MTP의 gemma4 layer는 더 작음). VRAM 극한 절약이 아니면 DFlash 우선.

## 실전 운영 팁 (댓글 누적)
[원본 §Comments →](2026-05-26_beellama_v020_dflash_3090_source.md#comments)

- **Context**: 단일 3090에서 **최대 150k** (moderate cache quant). long-context 출력 품질은 zai-lab 미학습 구간(>32k)에서 acceptance 저하.
- **16GB VRAM**: Qwen3.6-27B를 **Q3 / UD-Q3**로 압축 + `turbo3_tcq` KV → 들어감. IQ4_XS/iq3XXS 보고 사례 있음. **8GB**에서도 35B-A3B MoE DFlash ~40 tok/s 보고.
- **Tool calling (중요)**: v0.2.0 안정. 단 **V cache `q4_0`는 tool-call 문법을 깨뜨림**(precision loss) → **최소 `q4_1`** 사용. lazy grammar tool-call 구간은 drafting 일시 중단(specdec 미적용) — 긴 edit이 손해.
- **알려진 크래시**: `argmax.cu:557: GGML_ASSERT(K <= 32) failed` → **`--top-k 64`가 원인**(`--spec-dflash-cross-ctx 1024`도 연관). **top-k 20으로 낮추면 해결**, PR 머지됨(최신 main pull).
- **빌드**: CUDA 13.1 재빌드, `make -j 2`로 OOM 회피. Windows는 prebuilt 제공. 멀티 GPU `inconsistent sequence positions` 에러 — **멀티 GPU 미지원**(보고 기반 수정만).
- **정확도**: DFlash/MTP 모두 손실 없음 — drafter 출력 전부 target이 검증.

## 한계 & 포크 논쟁
[원본 §Comments →](2026-05-26_beellama_v020_dflash_3090_source.md#comments)

- 멀티 GPU 미지원, prefill 가속 없음, long-context acceptance 저하, 검증 모델 2종(Qwen/Gemma)뿐.
- **포크 계보**: bee-llama → buun-llama → TheTom/llama → [[llama.cpp]]. 단편화 비판 vs "포크는 R&D 레이어"(CatTwoYes +10: flash-attn·KV quant·spec-decode 모두 포크에서 시작). Anbeeld: mainline MTP/spec 아키텍처 곧 rebase 예정.
- TurboQuant + DFlash 조합이 핵심 매력(Rikers88, PyData Amsterdam 후원). dflash+ngram+copyspec **스택**은 초기 탐색 단계. MoE 전용 업데이트 별도 예정.
