# Qwen3.6 듀얼 RTX 3090 풀스택 서빙 — 25 → 283 tok/s

## TL;DR
[원본 §From 25 to 283 tok/s →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#from-25-to-283-toks-serving-qwen36-on-dual-rtx-3090s)

- 같은 듀얼 [[RTX 3090]] (24GB×2, [[Ampere]] SM 8.6, NVLink 없음, PCIe Gen4)에서 **두 가지 모델**을 단일 엔드포인트로 서빙. **모델 아키텍처가 토폴로지 선택을 뒤집은** 케이스 스터디.
- **Round 1 — Qwen3.6-27B 덴스 (하이브리드 attention + DeltaNet)**: 25 → **100 tok/s 단일 스트림 / 225 tok/s aggregate** (C=4). 비결: **TP=2 폐기 → 2 replicas + nginx least_conn LB** + [[MTP]] *n*=5 spec-decode + [[Genesis Patches|Genesis]] P67/P82/PN8 + FP8 KV + [[AutoRound]] INT4 (Lorbus 체크포인트, MTP head BF16 보존).
- **Round 2 — Qwen3.6-35B-A3B MoE (3B-active sparse)**: **283 tok/s aggregate** (C=4). 비결은 정반대 — **TP=2 + `--enable-expert-parallel` 단일 인스턴스**, **MTP 제거**(profile_run OOM + 공식적 net regress), `max-num-seqs=4` 천장(modular kernel workspace lock 버그).
- 핵심 교훈: **NVLink 없는 컨슈머 GPU에서는 PCIe NCCL all-reduce가 TP=2를 죽인다** — 단, MoE의 [[Expert Parallelism|expert-parallel]]은 dense TP 대비 NCCL 트래픽이 훨씬 적어 "TP=2 패배" 결론이 그대로 적용되지 않는다. **모델이 단일 카드에 들어가는가**가 replicate vs shard의 1차 결정자.

## 하드웨어 & 출발 가설
[원본 §The hardware →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#the-hardware)

- 2× RTX 3090 (24GB, SM 8.6, BW 936 GB/s) · Ubuntu 22.04.5 · 시작 드라이버 570.172.08 (CUDA 12.8 ceiling) → 최종 580.159.03 (CUDA 13.0 capable).
- [[Qwen3.6-27B]]는 **하이브리드 아키텍처** — softmax attention 레이어와 [[DeltaNet]]/GLA 선형 attention 레이어가 인터리브. 이 하이브리드성이 [[TurboQuant]] KV 압축을 *원천 차단*하는 트리거가 된다("attention + Mamba" 가드).
- 모든 27B 공식 체크포인트는 vision tower 포함 — "text" 변형이라도 멀티모달.
- 목표는 단순: **single endpoint, fastest tok/s for an agentic workload**.

## 최종 아키텍처 — single endpoint, 2 replicas, no NCCL
[원본 §Architecture — the final stack →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#architecture--the-final-stack)

```
client → nginx :8400 (least_conn) → vLLM :8500 (GPU 0)
                                  → vLLM :8501 (GPU 1)
                Model: Qwen3.6-27B AutoRound INT4 + Genesis + MTP n=3
```

- 두 vLLM replica **사이에 NCCL 없음** — 이게 전체 디자인의 핵심. agent의 동시 호출이 GPU 단위로 분산.
- `least_conn`은 round-robin 대비 hot-spotting 회피 (긴 응답이 특정 replica에 쌓이는 것 방지).
- 단일 엔드포인트 `:8400`에서 ~225 tok/s aggregate, **per-replica 컨텍스트 조율 없음**.

## 27B 덴스 — 단계별 throughput 진화
[원본 §The journey, stage by stage →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#the-journey-stage-by-stage)

| Stage | tok/s C=1 | TPOT (ms) | 핵심 변경 |
|---|---|---|---|
| 1. Eager (CUDAGraphs OFF, OOM 회피용) | 25.1 | 35.6 | TP=2, AWQ, BF16 KV — *기준선* |
| 2. + CUDAGraphs ON | 43.2 | 18.5 | `--gpu-mem-util` 0.92 → 0.83 + `--max-num-seqs 8`로 캡쳐 footprint 축소 → **dispatch 오버헤드 제거**, 1.7× |
| 3. + MTP *n*=3 spec-decode | 51.1 | 14.3 | Qwen3.6 내장 [[MTP]] head를 vLLM이 draft 모델로 사용 (3 토큰 예측 후 main 모델 1-pass 검증) |
| 4. AutoRound + Genesis P67/P82/PN8 (single GPU) | 60.6 | 12.2 | **Stage 4의 벽**(아래) 돌파 후 — TurboQuant 3-bit KV 가능 |
| 5. fast-chat preset (FP8 KV, mem-util 0.97) | 64.3 | 12.3 | KV가 병목이 아닐 때 TurboQuant 압축 오버헤드가 손해 → **FP8 KV로 회귀**가 더 빠름 |
| 6. + 4096-output amortization | 89.5 | 11.0 | 출력 길이 256 → 4096만 바꿔도 64 → 89 — **TTFT/cudagraph 컴파일 비용 희석** |
| 7. + MTP *n*=5 (single-GPU peak) | **100.3** | 9.7 | MTP=5에서 정점, MTP=6은 regress (draft acceptance 하락이 spec gain 초과) |
| 8. 2 replicas + nginx LB, C=4 agentic | **225.4** | 15.1 | TP=2 폐기 → 2 독립 replica + LB |

> 같은 듀얼 3090 하드웨어, 같은 prompt shape(1024 in / 256–4096 out)에서 측정한 leaderboard. 누적 9× throughput.

## Stage 4의 벽 — driver 580 강제 업그레이드 + Genesis 패치
[원본 §Stage 4: the wall →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#stage-4-the-wall--and-why-we-had-to-rebuild-from-the-foundation)

두 개의 roadblock이 동시에 hit:

**Roadblock A — CUDA/driver mismatch.** TurboQuant이 들어간 4월 15일 이후 모든 vLLM 이미지는 CUDA 12.9/13.0 빌드 + 드라이버 ≥ 575 요구. 570에서는 `Error 804: forward compatibility was attempted on non supported HW`로 worker 즉사. apt 패키지 충돌(NN-suffixed `nvidia-driver-570` ↔ `-580` 서브패키지) 때문에 일반 업그레이드 실패 — `apt-get -o Dpkg::Options::="--force-overwrite" install -y -f libnvidia-gl-580` + `apt autoremove --purge` + `dpkg --configure -a` + reboot으로 580.159.03 도달.

**Roadblock B — TurboQuant이 hybrid model을 거부.** `--kv-cache-dtype turboquant_k8v4` 시도 → `NotImplementedError: TurboQuant KV cache is not supported for hybrid (attention + Mamba) models`. Qwen3.6의 [[DeltaNet]] 선형 attention 레이어가 "Mamba-like"로 감지되어 boundary layer protection guard에 걸림.

**해법 — [[Sandermage]]의 [[Genesis Patches]] v7.14**: `P4 TurboQuant hybrid model support`가 정확히 이 가드를 풀어준다. + [[noonghunna]]의 reproducible-stack repo + **`Lorbus/Qwen3.6-27B-int4-AutoRound`** 체크포인트 (MTP head를 INT4 대신 BF16로 보존 → spec-decode draft 품질 보존). Genesis 부팅 시 GPU 자동 감지 후 self-recommend:

```
[Genesis GPU profile] detected: NVIDIA GeForce RTX 3090
  cc: (8, 6)  L2: 6 MB  BW: 936 GB/s  regime: bandwidth
[REC] P67  Multi-query verify kernel for spec-decode K+1   gain: +25-35%
[REC] P82  SGLang-style acceptance threshold OR-clause     gain: +8-12%
[REC] PN8  MTP/draft online-quant propagation              gain: 0% TPS, ~1-2 GiB VRAM
```

세 개 모두 켜고 60.6 tok/s. **GPU의 bandwidth-bound regime**에 맞게 패치가 자동 선택된 점이 핵심 — Genesis는 SM/L2/BW를 보고 권장을 바꾼다.

## MTP *n*=5 — 발표된 천장을 깨다 (89 → 100 tok/s)
[원본 §Stage 7: MTP n=5 →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#stage-7-mtp-n5--push-past-the-published-ceiling-100-toks)

vLLM 공식 문서와 fzbcwvv의 Medium 글 모두 **MTP depth >1은 같은 layer 재사용 → acceptance 하락**이라고 경고. 실측은 둘 다 보수적이었음:

| MTP depth | tok/s C=1 (4096 out) |
|---|---|
| 1 | 47.3 |
| 2 | 50.5 |
| 3 | 89.5 |
| 4 | 107.0 |
| 5 | **113.0 (5-prompt 이상값) / 100.3 (10-prompt 안정값)** |
| 6 | 105.2 (regress) |

5-prompt 평균은 cold-start TTFT 이상치로 113 tok/s까지 튀었고, 10-prompt warmed-up이 진짜 100. **"첫 사용 가능한 벤치는 항상 거짓말한다"**는 교훈. MTP=5가 local maximum, =6에서 draft acceptance 하락이 spec gain을 추월. **경고문보다 측정을 신뢰**가 결론.

## nginx LB > TP=2 — single endpoint 225 tok/s aggregate
[원본 §Stage 8: doubling throughput →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#stage-8-doubling-throughput-with-a-load-balancer-225-toks-aggregate)

핵심 측정: **TP=2 + Genesis tunings = 55-68 tok/s** (NCCL all-reduce per layer가 compute speedup를 잡아먹음). 2개 독립 replica는 NCCL 없이 **per-replica 100 tok/s 그대로 보존** → 더해서 ~225.

```nginx
events { worker_connections 4096; }
http {
    upstream vllm_pool {
        least_conn;
        server 127.0.0.1:8500;
        server 127.0.0.1:8501;
    }
    proxy_buffering off;          # SSE / streamed completions
    proxy_request_buffering off;
    proxy_read_timeout 900s;
    server { listen 8400;
        location / { proxy_pass http://vllm_pool;
                     proxy_http_version 1.1; proxy_set_header Connection ""; } }
}
```

**Concurrency curve**:

| C | Aggregate tok/s | TPOT | TTFT |
|---|---|---|---|
| 1 | 81.8 | 11.2 ms | 1.0 s |
| 2 | 154.0 (1.88×) | 11.9 ms | 0.84 s |
| **4** | **225.4 (2.75×)** | 15.1 ms | 1.44 s |
| 8 | 167.0 (큐 형성) | 20.4 ms | **19 s** |

Sweet spot **C=4 = replicas(2) × max-num-seqs(2)**. 그 이상은 replica 내부에서 큐가 쌓여 TTFT 폭발. agent가 2~4 parallel tool call을 fan-out하는 워크로드에 완벽하게 맞는 형태.

추가 디테일: dual-replica에서는 `--max-num-seqs` 1 → **2** (concurrency가 replica 수를 넘을 때 흡수), MTP는 5 → **3** (high concurrency에서 per-stream acceptance gain 감소), `--gpu-memory-utilization` 0.97 → **0.92** ([[GDN]] 선형 attention warmup 버퍼 OOM 회피).

## Round 2 — MoE 35B-A3B에서 거의 모든 가정이 뒤집힘
[원본 §Round 2: Qwen3.6-35B-A3B (MoE) →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#round-2-qwen36-35b-a3b-moe--different-model-different-bottlenecks)

`Qwen3.6-35B-A3B`는 **35B 총 파라미터, 3B active per token**의 sparse MoE — 128 experts 중 ~8개로 동적 라우팅. 27B 덴스의 결론들이 *이 모델에는 그대로 적용되지 않는다*:

- **decode가 자연 빠름** — token당 3B-equivalent 연산만 필요. 27B-덴스가 MTP *n*=5로 끌어내린 9.7 ms TPOT을 **MoE는 spec-decode 없이 9.95 ms** 달성.
- **bottleneck이 메모리, 연산이 아님** — 어떤 expert가 라우팅될지 예측 불가 → **모든 expert가 VRAM 상주 필수**. AWQ-INT4 체크포인트 24GB가 단일 3090(24GB)에 KV+활성화 포함해서 안 들어감. → **2-replica + LB 패턴 적용 불가**, 강제로 단일 인스턴스 `--tensor-parallel-size 2 --enable-expert-parallel`.
- **prefill이 훨씬 빠름** — attention 레이어는 dense, sparse는 FFN/expert만. 동일한 1024-token prompt에서 mean TTFT가 27B-dense **1.0 s → MoE 277 ms**.

| Stack | Hardware | 단일 stream | C=4 aggregate | TPOT | 승리 요인 |
|---|---|---|---|---|---|
| 27B-dense (2 replicas + LB, MTP n=3) | 2× 3090 | 100.3(alone) / 81.8(LB) | 225.4 | 9.7 / 15.1 ms | MTP=5 + Genesis P67/P82 + fp8 KV |
| **35B-A3B MoE (TP=2 + EP, no spec)** | 2× 3090 | 90.3 / 98.9 | **282.7** | 13.5 ms | sparse 3B-active + Genesis MoE 패치 (P31/P17/P18/P24/P72) |

**MoE가 aggregate에서 +25%, TTFT는 ~1/3**. 27B-덴스는 단독 single-stream에서만 우세 (100 vs 90). **agent 시스템에는 MoE가 더 나은 선택**.

MoE 전용으로 켜진 [[Genesis Patches]]도 다름: P31(MoE router fp32 softmax), P17/P18(Marlin MoE per-SM tuning, 3090 디폴트 config 부재 보완), P24(fused_moe overlay), P72(profile_run M cap, `max-num-batched-tokens > 4096` 언락) — 대부분 자동 적용, P72만 env 플래그로 opt-in.

## MoE에서 regress한 것 — MTP와 max-num-seqs 8
[원본 §What didn't work (and why) →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#what-didnt-work-and-why)

- **MTP spec-decode → profile_run OOM**: MoE에서 `--speculative-config '{"method":"mtp","num_speculative_tokens":3}'` 추가 시 GDN 선형 attention warmup OOM. MTP 레이어 자체가 expert를 포함(`mtp.layers.0.mlp.experts.*`)하기 때문에 depth-3 = **3× expert footprint during dummy_run**. [thc1006의 llama.cpp 19-config exhaustive 벤치](https://github.com/thc1006/qwen3.6-speculative-decoding-rtx3090) 결론과 일치: **Ampere + A3B에서 net speedup을 내는 spec-decode 변형 0개**.
- **`--max-num-seqs 8` → workspace-lock crash**: P72로 `--max-num-batched-tokens 8192 --max-num-seqs 8` 부팅은 가능하지만, 실제 concurrency 8 도달 첫 순간 `RuntimeError: Workspace is locked but allocation from 'modular_kernel.py:1084:_allocate_buffers' requires 448.06 MB, current size is 256.00 MB`. profile_run 시 pre-allocate한 워크스페이스가 실시간 확장 불가 — **vLLM 버그**, Genesis/하드웨어 무관. **`--max-num-seqs 4`가 안전 천장**, 그리고 마침 GPU 포화점이라 throughput 손해도 없음.

## 핵심 교훈 (cross-round)
[원본 §Lessons learned →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#lessons-learned)

- **NVLink 없으면 TP=2 ≠ 정답** — PCIe Gen4 NCCL all-reduce 비용이 의외로 큼. 2 replicas + LB가 단일/aggregate 모두 우세.
- **MoE에서는 다시 TP=2+EP가 우세** — `--enable-expert-parallel`은 GPU간 ~3B-equivalent weight 트래픽만 발생(dense TP는 full hidden state) → NCCL 부담이 dramatically 낮음. **"TP=2 패배" 결론은 dense PCIe 한정.**
- **단일 인스턴스 vs replicate는 모델이 카드에 들어가느냐로 1차 결정**: 27B INT4 14GB → fits → replicate. 35B MoE INT4 24GB → 안 들어감 → 강제 shard.
- **하이브리드 아키텍처는 최적화를 *조용히* 부순다** — TurboQuant 가드는 `NotImplementedError`로 친절히 알려주지만, 가드 없으면 garbage output. dense에서 효과 본 최적화(TurboQuant, MTP)가 MoE에서 regress하는 패턴 동일 — **항상 재측정**.
- **첫 벤치는 거짓말** — 5-prompt 평균은 cold-start TTFT 이상치로 흔들림. 항상 warm + re-run.
- **긴 출력은 모든 비용을 희석**: TTFT, cudagraph 재컴파일, prefix cache warmup, MTP acceptance 안정화. 워크로드 실제 길이로 측정.
- **vLLM 문서·Medium 글의 경고치는 보수적**: MTP depth >1 경고에도 불구하고 *n*=5에서 +30% gain. Sandermage Genesis와 noonghunna repo는 공식 문서보다 2주 앞섬 — **GitHub issue/Reddit/community repo가 cutting-edge LLM serving의 진실**.
- **Ubuntu 드라이버 업그레이드는 여전히 적대적**: NN-suffix nvidia 패키지 분할 + apt autoremove의 잘못된 판단 → `--force-overwrite`가 dpkg 충돌의 reliable escape valve.

## RTX 4090 / 5090 적용 가능성
[원본 §Will this work on RTX 4090 / 5090? →](2026-05-07_qwen36_dual_3090_25_to_283_tps_source.md#will-this-work-on-rtx-4090--5090)

- **RTX 4090 (Ada SM 8.9, 24GB)**: 같은 레시피 동작 예상, BW ~2× (1008 vs 936 GB/s) → tok/s 더 높을 것. Genesis P67은 hardware-conditional이므로 launch 시 권장 재확인. **dense 27B에서 LB 패턴 여전히 우세** (4090 NVLink 없음).
- **RTX 5090 (Blackwell SM 12.0, 32GB)**: native FP8/FP4 가속 → fast-chat fp8 KV 경로가 hardware-accelerated, TurboQuant도 재고 가치. 32GB 덕분에 `--max-model-len`/`--max-num-seqs` 여유, 양자화 없이도 운용 가능. vLLM nightly `-cu130` 태그 사용. **MoE 32GB 카드 1장에 fits → 듀얼 5090에서 2-replica LB 패턴 가능 → 500+ tok/s aggregate 가능성**(미측정, 단 아키텍처 장벽은 사라짐).
- **공통 watch-out**: GDN 선형 attention OOM은 GPU 클래스 무관(per-prompt scratch buffer 이슈), upstream kernel fix 전까지 `--max-num-seqs 2` 유지. 5090은 driver 580도 너무 옛날일 수 있음 — Blackwell-specific fix가 590+에 있음.
