# PAB × MCP × 모델 사이즈 — 의사결정 기록

> 📌 PROJECT 노트 — 2026-05-27 대화에서 도출된 아키텍처 의사결정의 스냅샷. 외부 원문 없는 내부 종합이므로 SOURCE 짝 없음.

## 배경 — 현재 PAB-v3 구조

채팅 1회에 서로 다른 **연결 2종**이 섞여 동작:

```
[채팅 앱 = 오케스트레이터]
  (A) Vector/RAG 검색 (앱이 직접)  → 지식 docs          ← 지식 연결
  (B) [docs + 질문] 조립 → ollama xxx:xxx/v1/chat/completions  ← 추론 연결
      → 로컬 LLM 추론 출력
```

- **(A) 지식 연결** = "무슨 자료를 줄까" (RAG/Vector).
- **(B) 추론 연결** = "그 자료로 답을 만든다" (ollama `/v1/`, OpenAI 호환 추론 API).
- 둘은 다른 레이어인데 한 함수에 묶여 하나처럼 보임.

## 핵심 결정 1 — MCP는 (B) 추론을 대체하지 않는다

가장 흔한 오해. **MCP ≠ 추론 프로토콜.**

| 레이어 | 역할 | MCP 전환 시 |
|---|---|---|
| `/v1/chat/completions` | 추론 (텍스트→텍스트) | **그대로 유지** |
| MCP | 도구/데이터 호출 표준 배선 | (A) 지식 연결을 여기로 |

→ MCP를 도입해도 ollama `/v1/` 호출은 남는다. MCP는 그 **위에 도구 레이어로 얹힌다**. 바꿀 수 있는 건 **(A) RAG/Vector 검색 → MCP 서버**뿐. 상세: [[2026-05-27_mcp_spec_2026_07_28_rc|MCP 스펙 2026-07-28 RC]].

## 핵심 결정 2 — 제어권 이동: 앱 주도 RAG → LLM 주도 도구 호출

"동일 작동"이 아니라 **제어 흐름이 뒤집힌다.**

| | 현재 RAG | MCP 도구 방식 |
|---|---|---|
| 검색 결정 | **앱** (항상 1회 선검색) | **LLM** (0~여러 회, 조건부) |
| LLM 역할 | 수동 (조립된 프롬프트 완성) | 능동 (필요 시 도구 호출) |
| `/v1/` 추론 | 사용 | 동일 사용 |
| 검색 코드 | 앱에 박힘 | 독립 MCP 서버로 분리 |

- MCP는 판단하지 않는다 — **LLM이 "검색할지" 결정**, MCP는 그 호출을 운반.
- 이득: 검색 로직을 앱에서 떼어 **언어무관 표준 서버로 분리**(재사용). RC의 **stateless** 덕에 평범한 웹서비스로 배포 가능.
- 비용: 호스트에 **tool-call 루프** 구현 + 로컬 LLM **tool-calling 품질** 확보.

## 핵심 결정 3 — tool-call은 지식 부족만 메운다 (추론 부족은 못 메움)

위임 구조의 함정:

| 모델 약점 | tool-calling이 보완? |
|---|---|
| **지식 부족** (모르는 사실) | ✅ 외부에서 가져옴 |
| **추론 부족** (언제/뭘 부를지 판단) | ❌ tool-calling 자체가 요구하는 능력 |

→ "도구를 잘 쓰는 것"이 곧 추론 능력. **너무 약한 모델은 도구를 쥐여줘도 오케스트레이션 실패** → 위임이 오히려 불안정. 약한 모델 + 전면 위임은 금물.

## 핵심 결정 4 — 위임은 0/1이 아니라 스펙트럼

MCP 도입 ≠ 전부 LLM 위임. 모델 역량에 맞춰 설계로 강제 정도 조절:

```
앱이 강제 ◄──────────────────────────────► LLM 전권
(기존 RAG)                                (순수 에이전트)

 강제 검색      라우터 앞단 판정     1차 강제+이후 선택      완전 위임
 (약한 모델)    (분류기)            (하이브리드, 권장)      (강한 모델)
```

약한 로컬 모델 보강 전략:
1. **1차 검색 강제 + 추가 검색만 위임** — 환각 바닥 방어 (권장 기본값).
2. **라우터/분류기 앞단** — "검색 필요?"를 작은 모델/규칙이 먼저 판정, 메인 LLM 부담 경감.
3. **constrained decoding / grammar** — tool-call 문법 강제로 "깨진 호출" 차단. → [[2026-05-26_beellama_v020_dflash_3090|BeeLlama]] 경고 직결: tool-call 시 **KV cache 최소 `q4_1`**(`q4_0`은 문법 깨짐), grammar 처리 검증 필수.
4. **tool description 정교화** — 약한 모델은 설명 품질에 민감.

## 핵심 결정 5 — 모델 사이즈: 로컬 vs 엔터프라이즈

엔터프라이즈 모델(GPT/Opus/Sonnet/Gemini)은 tool-call 타이밍 판단이 우수 → 위 보강 설계를 덜 해도 됨. **단 PAB 맥락에선 공짜가 아님:**

| 축 | 로컬(Qwen3.6 등) | 엔터프라이즈 API |
|---|---|---|
| **데이터 주권** | 개인 지식 외부 유출 없음 (PAB 존재 이유) | 외부 전송 → 캐시·로그·학습 잔존 위험 |
| 비용 | 전기료 | tool-call 왕복마다 토큰 과금 (agentic 루프 = 폭증) |
| 오프라인/지연 | 망 없이 저지연 | 망 의존 |
| tool-call 판단 | 약함 → 설계 보강 필요 | 우수 (단 over-call·환각 0은 아님) |

→ **PAB = Personal AI Brain**. 로컬 호스팅(3090, [[2026-05-07_qwen36_dual_3090_25_to_283_tps|Qwen3.6 듀얼 3090]])을 택한 이유 자체가 데이터 주권·비용. 엔터프라이즈 API로 가면 PAB 존재 이유와 충돌. **엔터프라이즈 모델은 설계를 게으르게 해줄 뿐, 설계를 없애주지 않는다.**

## 결론 — PAB의 본질 과제

> **"로컬 약한 모델로도 쓸 만한 tool-call 오케스트레이션을, 위임 스펙트럼 + 설계 보강으로 만든다."**

- 추론 배선(`/v1/`)은 불변, (A) 지식 연결만 MCP 도구로.
- 약한 모델 → 하이브리드(1차 강제 + 이후 위임) + grammar 강제 + 라우터.
- 모델 사이즈는 데이터 주권/비용 제약 하에서 선택 — 로컬 우선이 PAB 기본 노선.
