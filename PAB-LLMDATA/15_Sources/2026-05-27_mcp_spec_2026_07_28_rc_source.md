# The 2026-07-28 MCP Specification Release Candidate

**Date:** May 21, 2026
**Authors:** David Soria Parra (Lead Maintainer), Den Delimarsky (Lead Maintainer)
**Reading Time:** 9 minutes

## Introduction

The release candidate for MCP version 2026-07-28 introduces "the largest revision of the protocol since launch." Key additions include a stateless protocol architecture, first-class extensions framework, authorization improvements aligned with OAuth/OpenID Connect standards, and a formal feature lifecycle policy.

## A Stateless Protocol

The headline change removes the `initialize`/`initialized` handshake and `Mcp-Session-Id` header that previously required sticky session routing. Instead, requests become self-contained with protocol version and client info traveling in `_meta` fields on every call.

**Before and after:**
- Old model (2025-11-25): Required `initialize` handshake returning session ID; each subsequent request pinned to specific server instance
- New model (2026-07-28): Self-contained requests carrying client info in `_meta`; any server instance can handle any request

**Before example:**
```
POST /mcp HTTP/1.1
Mcp-Session-Id: 1868a90c-3a3f-4f5b
```

**After example:**
```
POST /mcp HTTP/1.1
MCP-Protocol-Version: 2026-07-28
Mcp-Method: tools/call
Mcp-Name: search
```

**Impact:** "A remote MCP server that previously needed sticky sessions, a shared session store, and deep packet inspection at the gateway can now run behind a plain round-robin load balancer."

**Stateless protocol, stateful applications:** The protocol no longer manages session state, but applications can implement explicit handles (like `basket_id`) passed between tool calls—making state visible to the model rather than hidden.

**Server-to-client requests, restructured:** Two SEPs restructure mid-call prompts:
- SEP-2260: Server-initiated requests only allowed during active client request processing
- SEP-2322: Multi Round-Trip Requests replace SSE streams with `InputRequiredResult` returning `inputRequests`, `requestState`, and allowing client retry with `inputResponses`

**Routable, cacheable, traceable:**
- SEP-2243: Requires `Mcp-Method` and `Mcp-Name` headers for gateway routing without body inspection
- SEP-2549: List/resource results carry `ttlMs` and `cacheScope` (modeled on HTTP `Cache-Control`)
- SEP-414: W3C Trace Context propagation documented for distributed tracing correlation

## Extensions Become First-Class

SEP-2133 formalizes the extensions framework: extensions use reverse-DNS identifiers, negotiate through capability maps, live in separate `ext-*` repositories with delegated maintainers, and version independently.

**MCP Apps (SEP-1865):** Servers ship interactive HTML interfaces rendered in sandboxed iframes. Tools declare UI templates ahead of time for prefetching and security review. UI actions use the same JSON-RPC protocol as direct tool calls, maintaining audit trails.

**Tasks graduates to an extension:** Tasks graduated from experimental core feature (in 2025-11-25) to official extension due to production redesign needs. The new lifecycle allows servers to answer `tools/call` with task handles; clients drive execution with `tasks/get`, `tasks/update`, and `tasks/cancel`. Task creation is server-directed; `tasks/list` is removed as unsafe without sessions.

**Migration requirement:** Code targeting 2025-11-25 experimental Tasks API requires migration to new lifecycle.

## Authorization Hardening

Six SEPs align authorization with OAuth 2.0 and OpenID Connect practices:

- SEP-2468: Clients must validate `iss` parameter per RFC 9207
- SEP-837: Clients declare OpenID Connect `application_type` during Dynamic Client Registration
- SEP-2352: Credentials bind to issuing authorization server; re-registration required for server migration
- SEP-2207: Documents refresh token requests from OpenID Connect servers
- SEP-2350: Clarifies scope accumulation during step-up
- SEP-2351: Clarifies `.well-known` discovery suffix

## Roots, Sampling, and Logging Are Deprecated

Three features deprecated under new lifecycle policy (SEP-2577):

| Feature | Replacement |
|---------|-------------|
| Roots | Tool parameters, resource URIs, or server configuration |
| Sampling | Direct LLM provider API integration |
| Logging | stderr for stdio transports; OpenTelemetry for structured observability |

These are annotation-only; methods/types continue working for twelve months minimum before removal.

## Full JSON Schema 2020-12 for Tools

SEP-2106 lifts tool `inputSchema` and `outputSchema` to full JSON Schema 2020-12 support. Input schemas retain `type: "object"` root constraint but allow composition (`oneOf`, `anyOf`, `allOf`), conditionals, and references. Output schemas unrestricted.

**Error code change (SEP-2164):** Missing resource errors change from custom `-32002` to JSON-RPC standard `-32602` Invalid Params. Clients matching literal `-32002` require updates.

## How the Protocol Evolves From Here

Three governance SEPs establish breaking-change-free evolution:

1. **Feature Lifecycle Policy:** Features have Active→Deprecated→Removed stages with minimum twelve-month deprecation windows
2. **Extensions Framework:** New capabilities ship as opt-in extensions, stabilizing before (if ever) entering the specification
3. **Conformance Requirement (SEP-2484):** Standards Track SEPs cannot reach Final status without matching conformance suite scenarios; SDK tier system scores official SDKs against this suite

## Release Timeline and Validation

- **RC Lock:** May 21, 2026
- **Final Specification Publication:** July 28, 2026
- **Validation Window:** Ten weeks for SDK maintainers and implementers
- **Tier 1 SDK Expectation:** Support shipped within window

Resources: Draft specification and changelog available at modelcontextprotocol.io; issues reported to specification repository GitHub; implementation questions routed to Working Group Discord channels.

## Looking Ahead

The release provides "the foundation we expect it to grow on for a long time" through stateless protocol infrastructure, extensions framework enabling independent capability timelines, and lifecycle policies protecting implementer investments.
